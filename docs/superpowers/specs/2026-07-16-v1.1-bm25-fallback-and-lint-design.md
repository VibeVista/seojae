# v1.1 — BM25 Search Fallback + Deterministic Wiki Lint — Design Spec

- **Date:** 2026-07-16
- **Status:** Approved (brainstorm 2026-07-16, both sections approved as-is)
- **Scope source:** v1.1 brainstorm — themes A (onboarding friction) and
  C (wiki quality automation) selected; items A1 (BM25 fallback) and
  C1 (`tools/lint.py`) chosen; BM25 integration mode: **auto-fallback**
  (not an extension-slot replacement).

## Context

Seojae v1.0.0's biggest onboarding barrier is the search stack: installing
`chromadb` + `sentence-transformers` pulls in torch, and the first reindex
downloads a ~470MB embedding model. New users cannot try the wiki loop
(ingest → query) without minutes of setup.

Separately, the Lint workflow is entirely LLM-driven. Its mechanical checks
(broken wikilinks, orphans, frontmatter validity) are deterministic and should
not consume LLM effort — and cannot run in CI today.

## Goals

1. `git clone` → working search in seconds, with zero heavy dependencies.
2. A deterministic `tools/lint.py` usable by humans, the LLM Lint workflow,
   and CI.

## Non-goals

- Replacing the chromadb backend (it remains the recommended full experience).
- New extension-slot mechanics — the fallback lives inside `tools/search.py`;
  `extensions/search-chromadb.md` keeps `provides: search-backend` and its
  command tokens are unchanged.
- Semantic lint checks (stale info, missing pages) — those stay LLM-side.
- Morphological analysis for Korean — CJK bigram tokenization is sufficient
  at personal-wiki scale.

## Feature 1: BM25 auto-fallback search

### Behavior

- `tools/search.py` attempts `import chromadb` (lazy, as today). On
  `ImportError` it switches to the BM25 backend and prints one stderr notice:
  the active backend and the upgrade path
  (`pip install -r requirements.txt` + `--reindex`).
- New flag `--backend {auto,chroma,bm25}`, default `auto`. `chroma` without
  chromadb installed → error exit 1 with install hint. `bm25` forces BM25 even
  when chromadb is available (useful for tests and offline work).
- CLI contract is identical across backends: `--query/--add/--reindex/--top/
  --index-path/--wiki-path`, output `<path> [score: X.XX]`, exit codes
  0 (success) / 1 (error) / 2 (index missing). WIKI_SCHEMA workflows and the
  search-chromadb extension require no command changes.

### Implementation

- Pure-Python BM25 Okapi (k1=1.5, b=0.75), no new dependencies. Lives in
  `tools/bm25.py` (importable unit), wired through `tools/search.py`.
- **Tokenizer:** lowercase; Latin/numeric word tokens via regex; CJK
  characters additionally emitted as character bigrams (plus single chars for
  1-char runs). This gives practical Korean/Japanese/Chinese matching without
  a morphological analyzer.
- **Index:** JSON document at `search-index/bm25.json` (already gitignored):
  tokenized term frequencies per page, document lengths, page metadata
  (path, title). Rebuilt by `--reindex`; `--add` upserts a single page.
  Missing/corrupt index file → exit 2 (same semantics as the chroma path,
  triggering the schema's index.md fallback + Reindex advice).
- Indexed text = frontmatter title/tags/aliases + body, same
  frontmatter-required skip rule as the chroma backend.

### Score scale

BM25 raw scores are unbounded, but the Query workflow's "below 0.5 → also
scan index.md" rule assumes a bounded score. BM25 results are normalized
relative to the top hit (top result = 1.00) before output. Consequence: any
non-empty result set passes the threshold; an empty result set triggers the
existing fallback. Document this in the search-chromadb extension's
counterpart note and CLI reference.

### Install path

- `requirements.txt` (full stack) unchanged.
- New `requirements-lite.txt`: `PyYAML`, `pytest` only.
- README Quick Start and WIKI_SCHEMA Setup gain the two-tier path:
  lite install (seconds, BM25) now; upgrade to semantic later with
  `pip install -r requirements.txt` + Reindex.

## Feature 2: `tools/lint.py`

### Checks (all deterministic, no LLM)

| # | Check | Detail |
|---|-------|--------|
| 1 | Broken wikilinks | `[[X]]` in `wiki/` pages matching no page title, alias, or filename stem (`index.md` entries are covered by check 5, not this check) |
| 2 | Orphan pages | zero inbound wikilinks from other wiki pages (an `index.md` listing does not count as a link) |
| 3 | Frontmatter validity | parseable YAML; required fields `title/type/created/updated`; `type` ∈ entity/concept/source/synthesis; dates `YYYY-MM-DD`; `tags`/`aliases`/`sources` are lists |
| 4 | Filename uniqueness | stems unique across all `wiki/` subdirectories (Obsidian shortest-path rule) |
| 5 | index.md sync | wiki pages missing from `index.md`; `index.md` entries pointing at nonexistent pages |

Wikilink matching must mirror the search tool's page model: a link resolves
against page filenames (kebab-case stems), frontmatter `title`, and `aliases`.

### Interface

- `python tools/lint.py [--wiki-path wiki/] [--index-md index.md] [--json]`
  (flag named `--index-md`, not `--index-path`, to avoid collision with
  search.py's ChromaDB directory flag)
- Default output: human-readable report grouped by check, one issue per line
  with `path: message`.
- `--json`: machine-readable list of `{check, path, message}` for the LLM
  workflow and CI annotations.
- Exit codes: `0` clean, `1` issues found, `2` usage error (bad path).
- Orphan-page findings are warnings by default (new wikis legitimately have
  few links); `--strict` promotes warnings to exit-code-1 issues. Checks 1,
  3, 4, 5 are always issues.

### Integration

- **WIKI_SCHEMA Lint workflow:** new step 1 — run `tools/lint.py --json`
  first; the LLM consumes its findings, then performs semantic checks
  (stale info, missing pages, cross-reference and growth suggestions).
  Renumber the remaining steps.
- **CI:** add a lint step to `.github/workflows/ci.yml` running
  `python tools/lint.py` against the repo's example wiki (keeps shipped
  example content healthy).
- **AGENTS.md:** mirror the WIKI_SCHEMA Lint workflow change (inlined copy).

## Error handling

- BM25 index corrupt/unreadable → exit 2 with Reindex advice (never crash).
- lint.py on a page with unparseable frontmatter reports it under check 3 and
  continues (one bad page must not abort the run).
- Both tools remain stdlib+PyYAML-only so the lite install always runs them.

## Testing

- `tests/test_bm25.py`: tokenizer (Latin, Korean bigram, mixed), scoring
  order, relative normalization, index round-trip, exit codes, `--backend`
  forcing, chromadb-absent simulation (import shim), output format
  `<path> [score: X.XX]`.
- `tests/test_lint.py`: fixtures per check (broken link, orphan, bad
  frontmatter, duplicate stem, index drift), `--json` shape, exit codes,
  `--strict`.
- Existing `tests/test_search.py` must pass unchanged (chroma path intact).
- CI benefit: BM25 and lint tests are model-free — they run fully under
  `SKIP_MODEL_TESTS=true`.
